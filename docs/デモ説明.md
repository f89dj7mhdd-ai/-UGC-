# デモ説明トークスクリプト（最終面接用）

口頭説明の進行台本です。所要 **5〜7分** を想定。丸暗記ではなく、各パートの「言いたいこと」を自分の言葉で話すための骨子として使ってください。

---

## 0. 事前準備（話す前にやっておく）

- ターミナルでプロジェクト直下に `cd` 済みにしておく
- `python -m youtube_ugc.cli "高山" --sample --out report.html` を一度実行し、レポートが開ける状態にしておく（本番で固まらないための保険）
- ブラウザでレポートを開いておき、企画書も別タブで開く

## 1. 導入（30秒）

> 「観光地名を入れると、YouTube上のUGCを“どの市場の人に、どの話題が効いているか”で可視化して、自治体担当者がインバウンド施策の手がかりを得るツールを作りました。まず、この課題で私が一番重要だと考えた着眼点からお話しします。」

## 2. 着眼点（1分）— ここが評価の核

> 「最初に考えたのは、“この課題で本当に難しいのは分析ロジックではなく、データが取れるかどうかだ”ということです。SNSのAPIは年々厳しく、第三者の投稿を合法的に・現実的なコストで取れるかが成否を分けます。だからデータ取得の実現性を出発点にしました。」

> 「同時に、当初の要件には穴があると気づきました。たとえば“インプレッション分析”とありますが、インプレッションは投稿者本人しか見られず、第三者UGCでは取得できません。そこで、要件をそのまま実装するのではなく、穴を埋めてから実装する方針にしました。」

## 3. 取捨選択（1分）

> 「主要SNSのAPIを費用と取得しやすさで比較しました。Xは無料枠が廃止され従量課金で高額、TikTokは非営利・学術に限定され商用は使えません。一方YouTubeは無料で審査も軽く、観光動画も豊富です。そこでMVPはYouTube単独に絞りました。“やりたくても取れない・高い”ものを最初から外す判断が、実現可能性の設計だと考えています。」

## 4. デモ実演（1.5分）— 画面を見せる

実行して見せる、または既に開いたレポートを指しながら：

```bash
python -m youtube_ugc.cli "高山" --sample --out report.html --open
```

レポートを上から指して：

1. **現状サマリ** — 「収集24件、平均エンゲージメントや言語構成が一目で分かります。検索表記の欄に、地名を多言語化して集めたことが出ています。」
2. **市場別セグメント** — 「ここが肝です。言語＝主軸、地域＝従軸で分け、“中国語圏ではこの話題が伸びている”と市場ごとに比較できます。少数のセルには参考値フラグ、地域は自己申告ベースの推定値だと明示しています。」
3. **視聴者層と感情** — 「視聴者属性はAPIで取れないので、コメントの言語構成を視聴者側の代理にしています。サンプルでは“発信は日本語中心なのにコメントの約7割が外国語”＝海外の関心が発信より高い、という補正が見えます。感情分析でネガ要因の起点も拾います。」
4. **高反応パターン** — 「上位群に偏って多い特徴を、全体比の倍率（lift）で出しています。サンプルではショート動画が3.4倍偏在、など再現できる示唆になります。」
5. **示唆** — 「非専門の担当者がそのまま読めるよう、文章で次の手がかりを提示します。」

> 補足デモ（任意）：`--auto-aliases` で任意の観光地に多言語対応、`--no-aliases` を付けると日本語名のみで件数が激減するので、多言語化の効果を“前後比較”で見せられます。

## 5. 工夫した点（1分）

> 「特に4点。1つ目、取れないインプレッションを正直に捨てて指標を再定義したこと。2つ目、セグメント分割で“誰にとっての人気か”を切り分け、当初課題の“曖昧な評価”を解いたこと。3つ目、人気をliftで定量化したこと。4つ目、取れない“視聴者層”をコメントの言語構成で近似し、感情分析も入れたことです。」

> 「実データに当たって初めて見える問題にも向き合いました。たとえば“富山”と“toyama”で結果が全然違う＝地名の言語で偏ると気づき、多言語地名に対応。さらにWikidataだと“別府”が姓に化けたので、Wikipediaの言語間リンクに切り替えて解決しました。表現揺れはLLM（ローカルのOllama対応・無ければ辞書）で吸収しています。テストは36件。完璧ではないですが、調べて作り直し切る姿勢は示せたかと思います。」

## 想定質問と回答メモ

**Q. なぜYouTubeだけ？ Instagramの方が観光に強いのでは？**
→ Instagramはビジネスアカウント連携とアプリ審査が必要で、ハッシュタグ検索も週30件に制限されます。MVPで素早く検証するにはYouTubeが最短。Instagramは拡張フェーズに位置づけています。

**Q. エンゲージメント率の分母を登録者数にした根拠は？**
→ リーチやインプレッションは第三者UGCでは取得できないため、安定して取れる登録者数を分母にしました。登録者数を非公開にしているチャンネルは率を出さず絶対数で扱う、という例外も設けています。

**Q. 地域判定はどれくらい正確？**
→ 正直、高くありません。チャンネルの国情報は自己申告で空欄も多く、“発信者の地域”であって“視聴者の地域”ではありません。だから地域は従軸の推定値とし、信頼度フラグを付けて過信させない設計にしています。判断は言語を主に行います。

**Q. UGCの動きが本当にインバウンド増につながる？**
→ 因果は弱いと認識しています。UGC指標は“先行指標”、実来訪は“遅行指標”として分け、後フェーズで宿泊統計や人流データと突合して関係を検証する設計にしています。

**Q. この後どう発展させる？**
→ 競合観光地との比較、検索・閲覧トレンド（Google Trends・季節性）、Google Placesクチコミの併用、外部指標（宿泊統計・人流）との突合での成果検証、の順で考えています。

**Q. AI/LLMはどこで使う？**
→ 話題分類とコメント感情分析に使っています。表現揺れ・多言語を吸収するためで、ローカルのOllamaに対応（キー不要）、Anthropic/OpenAIキーがあればそちらを優先、無ければキーワード/辞書に自動フォールバックします。誰でも動かせることを優先しました。

**Q. 視聴者層が取れないのに、どう「視聴者」を見ているの？**
→ コメントの言語を判定して視聴者側の市場の代理指標にしています。発信者言語と並べて比較することで、「発信は日本語でも視聴者は海外」といったギャップが見えます。あくまで近似で、年齢・性別までは取れないと正直に位置づけています。

**Q. 任意の観光地に対応できる？地名はどうやって多言語化？**
→ 主要地は内蔵辞書、それ以外はWikipediaの言語間リンクで各言語版タイトル（＝その言語の地名）を取得します。最初はWikidataで試しましたが同名の姓に化ける曖昧性があり、Wikipedia記事の言語カバレッジで選ぶ方式に作り直しました。取得失敗時は辞書・地名そのものにフォールバックします。
